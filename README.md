# SPAN

[![Paper](https://img.shields.io/badge/Paper-CVPR%202026-b31b1b?logo=adobeacrobatreader\&logoColor=white)](https://openaccess.thecvf.com/content/CVPR2026F/papers/Wu_Learning_Spatial-Preserving_Hierarchical_Representations_for_Digital_Pathology_CVPRF_2026_paper.pdf)
[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)

Official implementation of **SPAN: Learning Spatial-Preserving Hierarchical Representations for Digital Pathology**.

SPAN is two things at once:

* the **reference model** from the CVPR 2026 paper, and
* a **reusable framework for sparse hierarchical modeling** on whole-slide images — a set of interchangeable sparse blocks (down/upsampling, window/global attention) and a declarative network spec, so you can assemble a task-specific network instead of being locked into one fixed architecture.

Out of the box it supports:

* slide-level classification
* patch-level segmentation
* slide-level survival analysis
* custom sparse hierarchical models assembled through `src.span` and the `slide_configs` spec

Unlike standard MIL models that treat a slide as an unordered bag of patch features, SPAN preserves the spatial layout of WSI patches and builds sparse hierarchical representations directly on the patch grid.

## Overview

Whole-slide images are extremely large, but tissue patches usually occupy only a sparse subset of the full slide canvas. SPAN is designed to model this sparse spatial structure directly.

A typical SPAN workflow is:

```text
raw whole-slide images
  -> patch extraction with global WSI coordinates
  -> patch feature extraction
  -> SPAN training / evaluation
```

The recommended preprocessing workflow is:

```text
raw WSIs
  -> modified CLAM patch extraction
  -> PatchPreprocess feature extraction
  -> SPAN training / evaluation through data_root
```

## Data preparation

SPAN does **not** read raw WSI files directly. It consumes pre-extracted slide-level feature files that contain both patch features and patch coordinates.

We recommend preparing these feature files with [PatchPreprocess](https://github.com/wwyi1828/PatchPreprocess). PatchPreprocess consumes patch folders generated by the modified CLAM preprocessing code at [wwyi1828/CLAM](https://github.com/wwyi1828/CLAM). This CLAM branch keeps patch coordinates aligned to a global 224-step WSI grid, which is the default coordinate convention expected by SPAN.

### Why the coordinate convention matters

SPAN is **not** an unordered-bag MIL model. Every patch is placed on an integer grid, and that grid is what the whole architecture is built on:

* **Hash indexing.** SPAN sorts and indexes patches with an integer space-filling key (`x * (y_max + 1) + y`). This only produces a clean, collision-free index when coordinates are exact non-negative integers on a regular grid.
* **Index-based sparse up/downsampling.** Sparse strided convolutions and patch-merge/inverse blocks build their rulebooks by integer-dividing coordinates (e.g. `coord // stride`). Non-divisible or jittered coordinates break the parent/child correspondence between resolution levels.
* **Sparse window attention.** Swin-style windowing partitions the grid into fixed `2 * window_size` blocks. Window assignment is computed from integer coordinates, so a misaligned grid silently scatters patches into the wrong windows.

Concretely, the loader normalizes coordinates as `pos = (raw_coords / 224).int()` (see `normalize_coords` in `tasks/vision/shared/data.py`). The integer cast is **truncating, not rounding**:

| Source | Example raw `cords` (pixels) | After `/224 → int` | Result |
| --- | --- | --- | --- |
| SPAN / modified CLAM | `(0, 0), (224, 0), (448, 224)` | `(0, 0), (1, 0), (2, 1)` | exact, lossless grid |
| Stock CLAM (overlap / odd patch size / level-0 offsets) | `(0, 0), (240, 0), (476, 230)` | `(0, 0), (1, 0), (2, 1)` | grid distorted, neighbors collapsed |

With the modified CLAM branch, raw coordinates are exact multiples of 224, so the conversion is lossless and the grid round-trips perfectly. Stock CLAM coordinates depend on patch size, overlap, and the WSI level offset, so they are generally **not** multiples of 224 — truncation then merges or shifts patches, corrupting the hierarchy that hash indexing, sparse sampling, and window attention all depend on.

If you use a custom preprocessing pipeline, make sure the saved coordinates are globally aligned WSI patch coordinates that are exact multiples of the patch step (or adjust `coord_div` to match your step), rather than arbitrary local or floating-point coordinates.

Each H5 feature file should correspond to one slide and contain at least:

```text
feats: patch features, shape [num_patches, feature_dim]
cords: patch coordinates, shape [num_patches, 2]
```

Optional fields include:

```text
label:   slide-level label
ratios:  patch-level annotation overlap ratios
caption: optional slide caption
```

The default loader assumes that `cords` are WSI pixel coordinates on a 224-step patch grid. These coordinates are converted to integer patch-grid coordinates internally. Arbitrary floating-point coordinates are not supported by default.

Custom preprocessing pipelines can also be used, as long as they produce compatible feature files with globally aligned patch coordinates.

Set `data_root` to the folder containing the prepared feature files:

```bash
python -m tasks.vision.slide.classification.main \
  data_root=/path/to/features
```

The exact dataset-specific folder names are defined by the corresponding config and dataset loader.

## Quick feature-file check

Before training, you can quickly inspect one H5 file:

```python
import h5py

path = "/path/to/features/example_slide.h5"

with h5py.File(path, "r") as f:
    print(list(f.keys()))
    print("feats:", f["feats"].shape)
    print("cords:", f["cords"].shape)
    print("first coords:", f["cords"][:5])
```

Expected fields include `feats` and `cords`. For the default 224 x 224 non-overlapping patch setup, coordinates usually follow a 224-step WSI grid.

## Usage

### Slide-level classification

```bash
python -m tasks.vision.slide.classification.main \
  data_root=/path/to/features
```

You can override the dataset and feature variant from the command line:

```bash
python -m tasks.vision.slide.classification.main \
  dataset=C16 \
  features_variant=R50 \
  data_root=/path/to/features
```

### Patch-level segmentation

```bash
python -m tasks.vision.patch.segmentation.main \
  dataset=C16 \
  features_variant=R50 \
  data_root=/path/to/features
```

For patch-level segmentation, each slide should provide patch-level labels or annotation overlap ratios, usually stored as `ratios` in H5.

### Slide-level survival analysis

```bash
python -m tasks.vision.slide.survival.main \
  data_root=/path/to/features
```

If the clinical metadata is stored separately, set:

```bash
export SPAN_CLINICAL_ROOT=/path/to/clinical_metadata
```

or pass it directly:

```bash
python -m tasks.vision.slide.survival.main \
  data_root=/path/to/features \
  clinical_root=/path/to/clinical_metadata
```

## Configuration

SPAN uses Hydra configs under:

```text
configs/
```

Common options can be overridden from the command line, for example:

```bash
python -m tasks.vision.slide.classification.main \
  dataset=C16 \
  features_variant=R50 \
  data_root=/path/to/features \
  logging.wandb.enabled=false
```

W&B logging is disabled by default and can be enabled explicitly:

```bash
python -m tasks.vision.slide.classification.main \
  logging.wandb.enabled=true
```

## Scope

SPAN is meant to be used as a small framework, not a single frozen network.

**What it provides**

* A library of interchangeable **sparse blocks** that all operate on the integer patch grid — down/upsampling (sparse strided conv, patch merging, pixel shuffle, transpose conv) and attention (window, shifted-window, and global-token variants).
* A **declarative network spec** via `slide_configs` (e.g. `"111-21"`: each digit is the number of transformer layers in a stage, and `-` separates the encoder from an optional decoder). Combined with the block-type knobs (`econvs_type`, `dconvs_type`, `trans_type`, `dtrans_type`, …), you can assemble an encoder-only or encoder–decoder network for your task without writing model code.
* Hydra configs and ready-to-run entrypoints for classification, segmentation, and survival.

**What it does not do (current scope)**

* It does **not** read raw WSIs or run patch/feature extraction — it consumes pre-extracted feature files (see [Data preparation](#data-preparation)).
* Coordinates must be globally aligned and integer-divisible; arbitrary floating-point coordinates are silently truncated (see [Why the coordinate convention matters](#why-the-coordinate-convention-matters)).

## Supported modules

SPAN is built from interchangeable sparse blocks selected through the model config (`configs/model/*.yaml`) and overridable on the command line. Because every block operates on the integer patch grid, downsampling and upsampling stay index-consistent across resolution levels.

### Down/upsampling (`econvs_type` / `dconvs_type`)

| Type | Role | Notes |
| --- | --- | --- |
| `dwpatchconv` / `patchconv` | Sparse strided (transpose) conv downsample | Depthwise (`dw`) or full conv; default encoder block |
| `patchmerge` | Sparse Swin-style patch merging | Concatenates a `stride × stride` neighborhood, then projects |
| `dwpatchinvs` / `patchinvs` | Sparse strided **upsample** (transpose conv) | Used in the decoder for segmentation |
| `pixelshuffle` | Sparse pixel-shuffle upsample | Sub-pixel expansion on the grid |
| `convnext` / `linear` / `projection` / `identity` | Same-resolution feature blocks | No spatial change |

Most blocks ship in two numerically equivalent implementations of the **same operation**: sparse (also addressable with the `s_` prefix) and dense (`d_` prefix, e.g. `d_patchmerge`, `d_pixelshuffle`). All default configs use the sparse path, which is faster at typical tissue sparsity. (Depthwise layers like `dwpatchconv` are an exception where dense is slightly faster, but the gap is small, so the default stays sparse for consistency.)

### Window / global attention (`trans_type` / `dtrans_type`)

| Mode | Description |
| --- | --- |
| `long` | Longformer-style: shifted + non-shifted window self-attention plus a global-token relay (default) |
| `long_noshift` | Same as `long` without the shifted window |
| `swin` / `tradswin` | Pure shifted/non-shifted window self-attention (Swin-style), no global tokens |
| `swin_global` | Window attention with an added global-token stage |

The window extent is controlled by `window_size` (windows span `2 * window_size` grid cells). Example overrides:

```bash
python -m tasks.vision.slide.classification.main \
  data_root=/path/to/features \
  model.econvs_type=patchmerge \
  model.trans_type=swin \
  model.window_size=4
```
